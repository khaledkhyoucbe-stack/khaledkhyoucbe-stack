القسم الثاني: اقتراح المعمارية Offline-First

2.1️⃣ رؤية المعمارية الجديدة:
──────────────────────────────

   التطبيق سيعمل بشكل مستقل عن الإنترنت مع مزامنة اوتوماتيكية عند الاتصال

   المستخدمون | القالب | Local DB | Sync Queue | Server API | Server DB
              ↓        ↓         ↓            ↓           ↓
   يعملون     قوالب   IndexedDB  Queue        REST API   SQLite/
   بلا       محسنة    (محلي)    للتغييرات            PostgreSQL
   إنترنت    + JS              المعلقة

2.2️⃣ المكونات الرئيسية:
───────────────────────

🔵 المستوى الأول: Layer 1 - الواجهة (UI Layer)

   ┌─────────────────────────────────────────────┐
   │         Updated Templates                    │

   │                                               │

القسم الثالث: تصميم تدفق البيانات (Data Flow Design)
════════════════════════════════════════════════════════════════════════════════

3.1️⃣ سيناريو العمل الطبيعي (Online):
──────────────────────────────────────

   المستخدم يضيف درجة جديدة:

   1. في الواجهة (UI):
      [المستخدم يملأ نموذج]
           ↓
      [JavaScript يتقاط الحدث]
           ↓
      
   2. في Local Storage:
      [حفظ محلي في IndexedDB]
      {
        "record_id": "temp_123456",
        "table": "grades",
        "action": "create",
        "data": {
          "student_id": 45,
          "subject_code": "MT",
          "grade": 85.5
        },
        "timestamp": "2026-04-06T10:30:00Z",
        "sync_status": "pending"
      }
           ↓
      [عرض التأكيد فوراً للمستخدم - تجربة سريعة]
           ↓
      
   3. في Sync Queue:
      [تصفية: يوجد اتصال إنترنت؟]
      
      نعم ✓:
        ↓ [فوراً أو بعد ثوان]
        
        4. الإرسال إلى الملقم:
           POST /api/sync/push
           Content:
           {
             "client_id": "user_laptop_123",
             "timestamp": "2026-04-06T10:30:05Z",
             "items": [
               {
                 "type": "create",
                 "table": "grades",
                 "data": {...},
                 "local_id": "temp_123456"
               }
             ]
           }
           ↓
           [انتظار رد الملقم]
           ↓
        
        5. معالجة الملقم:
           • التحقق من صحة البيانات
           • التحقق من الصلاحيات
           • فحص التضاربات
           • إدراج in main database
           • تعيين ID فعلي
           ↓
           رد ناجح:
           {
             "status": "success",
             "items": [
               {
                 "local_id": "temp_123456",
                 "server_id": 9876,
                 "sync_id": "abc123xyz",
                 "version": 1,
                 "timestamp": "2026-04-06T10:30:06Z"
               }
             ]
           }
           ↓
        
        6. في الجهاز (Client):
           • تحديث IndexedDB بـ server_id الجديد
           • تحديث sync_status إلى "synced"
           • تسجيل في sync_log
           ↓
           [تنبيه مرئي: "✓ تم المزامنة"]
      
      لا ✗:
        ↓ [بدون إنترنت]
        
        البقاء في Sync Queue:
        ← سيتم إعادة المحاولة عند الاتصال

3.2️⃣ سيناريو بدون إنترنت (Offline):
───────────────────────────────────────

   المستخدم بدون إنترنت:

   1. يفتح التطبيق
      ↓
   2. يحمل القالب والبيانات من Cache
      ↓
   3. يعرض رسالة مرئية: "⚠️ وضع بدون اتصال"
      ↓
   4. يسمح بـ:
      ✓ قراءة البيانات المخزنة محلياً
      ✓ إنشاء سجلات جديدة
      ✓ تحديث البيانات الموجودة
      
   لا يسمح بـ:
      ✗ تحميل بيانات جديدة
      ✗ عمليات تتطلب بيانات حية
      
   5. جميع العمليات:
      → تحفظ محلياً في IndexedDB
      → تضاف إلى sync_queue
      → يرى المستخدم علامة ⏳ (pending)
      
   6. عند عودة الاتصال:
      → كشف تلقائي
      → شروع المزامنة الاوتوماتيكية
      → عرض تقدم المزامنة
      → إزالة علامات pending عند النجاح

3.3️⃣ سيناريو المزامنة الثنائية (Bidirectional Sync):
──────────────────────────────────────────────────────

   ملخص النقل:
   
   [Local Changes]                [Server Changes]
         ↓                               ↓
   Sync Queue                    Backend Poll
         ↓                               ↓
   Push Changes (PUT/POST)    ← Pull Changes (GET)
         ↓                               ↓
   Server Process                Local Update
         ↓                               ↓
   Confirm Changes              Update IndexedDB
         ↓                               ↓
   Update Local Records      Notify User

القسم الرابع: استراتيجية المزامنة (Synchronization Strategy)
════════════════════════════════════════════════════════════════════════════════

4.1️⃣ حروف تشغيل المزامنة (Sync Triggers):
───────────────────────────────────────────

   🔔 المحفزات التلقائية:

   1. كشف الاتصال (Connection Detection):
      • عند عودة الإنترنت
      • Polling كل 30 ثانية
      • Event listener على "online/offline"

   2. فاصل زمني (Time-based):
      • كل 5 دقائق تلقائياً
      • أسرع عند وجود معلقات

   3. دعوة يدوية (Manual Request):
      • زر "مزامنة الآن"
      • بعد كل عملية حفريجة

   4. حجم الرتل (Queue Size):
      • عند وجود 10+ عناصر معلقة
      • فور اكتمال دفعة

4.2️⃣ خوارزمية المزامنة الأساسية:
──────────────────────────────────

   1. بدء المزامنة:
      ```
      IF internet_connected:
          GET pending_items from sync_queue
          IF pending_items.count > 0:
              batch_items = pending_items (up to 100)
              SEND batch_items to /api/sync/push
          ENDIF
      ENDIF
      ```

   2. معالجة الاستجابة:
      ```
      RECEIVE response from server:
          FOR EACH item IN response.items:
              IF item.status == "success":
                  Update local record with server_id
                  Set sync_status = "synced"
                  Delete from sync_queue
              ELIF item.status == "conflict":
                  Flag record as conflicted
                  Trigger conflict_resolver
              ELIF item.status == "error":
                  Log error
                  Retry with exponential backoff
              ENDIF
          ENDFOR
      ```

4.3️⃣ آلية الإعادة (Retry Mechanism):
──────────────────────────────────────

   لكل عملية معلقة:

   المحاولة الأولى:   فوراً
   المحاولة الثانية:  بعد 10 ثوان
   المحاولة الثالثة:  بعد 1 دقيقة
   المحاولة الرابعة:  بعد 5 دقائق
   المحاولة الخامسة:  بعد 30 دقيقة
   المحاولة السادسة:  بعد ساعة
   ثم: توقف ويطالب المستخدم بالتدخل

   Exponential Backoff Config:
   {
     "initial_delay": 10,      // ثوان
     "max_delay": 3600,         // ساعة واحدة
     "backoff_multiplier": 1.5, // اضرب كل مرة
     "max_retries": 6           // أقصى محاولات
   }

4.4️⃣ معالجة الأخطاء (Error Handling):
───────────────────────────────────────

   Errors هناك 3 أنواع:

   🔴 أخطاء دائمة (Permanent):
      • 400 Bad Request (بيانات خاطئة)
      • 401 Unauthorized (انقطع الجلسة)
      • 403 Forbidden (لا صلاحيات)
      
      الإجراء:
      → توقف الإعادة فوراً
      → عرض كخطأ للمستخدم
      → منح خيار يدوي (تصحيح/حذف)

   🟠 أخطاء متقطعة (Transient):
      • 500 Server Error
      • 503 Service Unavailable
      • Network Timeout
      
      الإجراء:
      → استمرار الإعادة مع backoff
      → عدم إرباك المستخدم
      → محاولة كل 5 دقائق

   🟡 أخطاء متضاربة (Conflict):
      • 409 Conflict
      • البيانات تغيرت على الملقم
      
      الإجراء:
      → اطلب بيانات الملقم الحالية
      → اعرض للمستخدم للاختيار
      → احفظ الاختيار محلياً

4.5️⃣ استراتيجية سحب البيانات (Pull Strategy):
─────────────────────────────────────────────

   سحب البيانات الجديدة من الملقم:

   الفاصل الزمني:
   • كل 5 دقائق تلقائياً
   • عند الاتصال مباشرة بعد التوصل
   • عند فتح كل صفحة جديدة

   نقطة نهاية الجلب:
   GET /api/sync/pull?last_sync=2026-04-06T10:25:00Z

   الاستجابة ستحتوي على:
   {
     "updates": [
       {
         "table": "grades",
         "action": "update",
         "record_id": 789,
         "data": {...},
         "version": 2,
         "timestamp": "2026-04-06T10:30:15Z"
       }
     ],
     "deletions": [
       {
         "table": "attendance",
         "record_id": 456
       }
     ]
   }

القسم الخامس: استراتيجية حل التضاربات (Conflict Resolution)
════════════════════════════════════════════════════════════════════════════════

5.1️⃣ ما هي التضاربات؟
─────────────────────

   حالات تحدث عندما:
   
   السيناريو الأول: تعديل متزامن (Concurrent Edit)
   ┌─────────────────┬──────────────────────────┤
   │ الجهاز A        │ الجهاز B                 │
   ├─────────────────┼──────────────────────────┤
   │ User opens      │                          │
   │ Grade = 80      │ User opens same record   │
   │                 │ Grade = 80 (نسخة قديمة too)
   │ Edits → 85     │                          │
   │ Saves locally   │ Edits → 90              │
    │ Syncs (85)     │ Saves locally            │
   │ Accepted ✓      │ Syncs (90)              │
   │                 │ CONFLICT! ⚠️             │
   │ Server has 85   │ But trying to save 90   │
   └─────────────────┴──────────────────────────┘

   السيناريو الثاني: حذف ثم تعديل (Delete then Modify)
   • أحد الأجهزة يحذف السجل
   • جهاز آخر يحاول تعديل نفس السجل
   • تضارب!

5.2️⃣ اختيار استراتيجية التحل:
───────────────────────────────

   لدينا خياران رئيسيان:

   الخيار 1️⃣: Last-Write-Wins (الكتابة الأخيرة تفوز)
   ───────────────────────────────────────────────
   
   الآلية:
   • التي جاءت آخراً تفوز
   • بسيط جداً
   
   المزايا:
   ✓ سهل التطبيق
   ✓ لا تعقيد
   ✓ سريع
   
   العيوب:
   ✗ قد تخسر بيانات
   ✗ لا يعكس النية الحقيقية
   ✗ غير آمن للبيانات الحرجة
   
   متى نستخدمه:
   • بيانات غير حرجة
   • الدرجات والحضور قد تكون مقبولة


   الخيار 2️⃣: Version-Based (قائمة على الإصدارات) ⭐ RECOMMENDED
   ──────────────────────────────────────────────
   
   الآلية:
   • كل سجل له version number
   • عند التعديل: version ← version + 1
   • عند المزامنة: قارن versions
   
   بنية إصدار:
   {
     "record_id": 789,
     "local_version": 2,
     "server_version": 3,
     "client_version": 2,
     "timestamp": "2026-04-06T10:30:00Z"
   }
   
   خوارزمية الحل:
   ```
   IF local_version == server_version - 1:
       // تحديث بسيط، لا تضارب
       Merge successfully
   ELIF local_version == server_version:
       // تعديل متزامن، نحتاج تحل
       Show user options:
         1. Keep local changes
         2. Accept server changes
         3. Manual merge
   ELIF local_version < server_version - 1:
       // الملقم تقدم كثيراً، قد يكون هناك تضارب
       Fetch server version
       Ask user to review
   ENDIF
   ```
   
   المزايا:
   ✓ حماية من فقدان البيانات
   ✓ اكتشاف التضاربات بدقة
   ✓ سماح للمستخدم باختيار
   ✓ آمن للبيانات الحرجة
   
   العيوب:
   ✗ أكثر تعقيداً
   ✗ يتطلب تخزين إضافي
   ✗ أبطأ قليلاً
   
   متى نستخدمه:
   • البيانات الحرجة (الدرجات، الرسوم)
   ✓ اختيارنا الأفضل


   الخيار 3️⃣: Operational Transformation (نادر جداً)
   ───────────────────────────────────────────────
   
   معقد جداً، غير ضروري لهذا المشروع

5.3️⃣ استراتيجيتنا الموصى بها:
──────────────────────────────

   🏆 سنستخدم: Version-Based مع Human Decision
   
   المنطق:
   • نظام أكاديمي يتطلب دقة
   • البيانات المالية والدرجات حساسة
   • المستخدمون سيتقبلون طلب التأكيد
   
   التنفيذ العملي:
   
   في قاعدة البيانات (كلا الطرفين):
   ```
   ALTER TABLE students    ADD COLUMN version INT DEFAULT 1;
   ALTER TABLE grades      ADD COLUMN version INT DEFAULT 1;
   ALTER TABLE attendance  ADD COLUMN version INT DEFAULT 1;
   ALTER TABLE fees        ADD COLUMN version INT DEFAULT 1;
   ... وهكذا لكل جدول
   ```
   
   في IndexedDB المحلي:
   ```
   {
     "record_id": "abc123",
     "table": "grades",
     "local_version": 2,
   ```

8.1️⃣ المخاطر التقنية:
──────────────────────

   🔴 المخاطر العالية:
   
   1. فقدان البيانات (Data Loss)
      المشكلة:
      • إذا حذفنا sync_queue بدون تزامن
      • انهيار المتصفح قبل الحفظ
      • حذف IndexedDB عن قصد
      
      التخفيف:
      ✓ نسخ احتياطية محلية متعددة
      ✓ تحذيرات قبل حذف
      ✓ حفظ في localStorage أيضاً
      ✓ تنبيهات قبل إغلاق التطبيق

   2. ازدواج البيانات (Data Duplication)
      المشكلة:
      • إرسال العملية مرتين بدون قصد
      • حفظ العملية محلياً وعلى الملقم
      • نقص معرفات فريدة
      
      التخفيف:
      ✓ معرفات فريدة (sync_id)
      ✓ deduplication على الملقم
      ✓ تتبع العمليات المرسلة

   3. عدم اتساق البيانات
      المشكلة:
      • نسخة محلية ≠ نسخة الملقم
      • تحديثات جزئية
      • فشل في التحديث الكامل
      
      التخفيف:
      ✓ نظام versioning قوي
      ✓ transactions على الملقم
      ✓ validation كامل

   🟠 المخاطر المتوسطة:
   
   1. مشاكل الأداء
      • IndexedDB بطيء مع بيانات ضخمة
      • المزامنة تستهلك bandwidth
      
      التخفيف:
      ✓ تقسيم البيانات (pagination)
      ✓ ضغط البيانات عند الإرسال
      ✓ caching ذكي

   2. مشاكل التوافقية
      • older browsers بدون IndexedDB
      • تقسيمات محمول vs desktop
      
      التخفيف:
      ✓ fallback إلى localStorage
      ✓ اختبار على browsers متعددة
